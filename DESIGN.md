# CheckIn Voice — Design

A running record of intentional design choices for the voice layer, with rationale and source citations. Written deliberately rather than by intuition; updated as David works through Cathy Pearl's *Designing Voice User Interfaces* and as questions arise.

`PLAN.md` covers scope and sequencing. This document covers *what we are building, and why*.

Each entry below should answer: the question being decided, what the references say, what we decide, and why. Citations to chapter and section so future revisits stay grounded.

## Decisions

### D1. Voice is state-based, not command-based

**Question.** Should the voice layer be designed as a set of stateless commands (the CLI mental model) or as a stateful conversation with memory of what just happened?

**What the references say.** Pearl frames VUI design fundamentally around dialog state and turn-taking, not isolated commands. Google's Conversation Design structures interaction as a flow with context persisting across turns. Apple's HIG for Siri assumes context-aware follow-up. The shared assumption across all three is that conversation, by nature, depends on memory of the prior turn.

**Decision.** Design the voice layer as a state machine. Each application state defines what utterances are expected, what TTS is appropriate, and what transitions exist out of that state. Anaphora ("read it", "the second one", "yes") is resolved against the current state, not parsed in isolation.

**Why.** A stateless voice layer forces every utterance to be self-contained ("read the email from Tony Smith about the project status update"), which is exhausting and unnatural. State-aware, the user can say "read it" because both sides know what "it" refers to. The web prototype was built on the wrong frame; this decision realigns to the conversational medium.

**Implications.** `VoiceDialog.swift` will be a state machine. The enumeration of application states is the central artifact of the rest of the design audit. Reply flow, disambiguation, confirmation, and error recovery all live as states with defined entry conditions and transitions.

### D2. CheckIn is multi-modal, not voice-only

**Question.** Should every interaction be voice-driven, or should voice be one of several modalities (touch, screen, keyboard) chosen per task?

**What the references say.** Pearl addresses multimodal design and cautions against forcing voice into tasks where another modality is faster or more accurate. Apple's HIG for Siri frames Siri as a complement to the on-screen UI, not a replacement. Google's Conversation Design treats voice and screen as channels that work together. NN/g writes on the discoverability problem with voice-only interfaces and the value of visual signifiers.

**Decision.** Use voice where it shines (eyes-off summary, hands-free reply dictation, quick triage commands) and fall back to touch and screen where they shine (list browsing, comparison, precise editing, ambient situations where speaking aloud is awkward). Each application state specifies which modalities it supports. Where multiple are supported, transitions between them are explicit and clean.

**Why.** Voice is bad at exactly the things screen-and-touch are good at, and vice versa. Forcing every interaction through voice would make the app slower and more error-prone for tasks the iPhone already handles well. The web prototype tried to voice-enable everything and lost the natural multimodal ergonomics of the device.

**Implications.** State enumeration must answer, for each state, what voice can do, what touch can do, what the screen shows, and how those combine. The mic button is an explicit modality switch. Visual signifiers (waveform, live transcript, listening indicator) matter; they teach the user what voice will accept and what it won't.

### D3. The voice layer maintains an explicit dialog context

**Question.** Beyond knowing what state we're in, what context does the system need to remember across turns to support natural conversation?

**What the references say.** Pearl emphasizes context tracking as a core requirement for any voice interface that aspires to feel conversational rather than command-driven. Dialog systems literature frames this as "dialog state tracking": the system keeps a structured representation of the conversation so far (entities mentioned, current focus, last utterance from each side) and uses that representation to interpret each new turn. Google's Conversation Design encodes this as context objects that scope the meaning of subsequent utterances. Apple's HIG for Siri encourages remembering recent input within a session.

**Decision.** Maintain an explicit `DialogContext` value alongside the state machine. The context tracks at least: the last summary read aloud (for back-references like "read the second one"), the currently focused entity (for "reply" / "read it" / "send him a message"), the most recent system utterance (for "repeat that"), and a short rolling history of user turns (for "no, the other one" style self-corrections). Resolved entities (the specific contact the user means by "Tony") are stored in the context, not re-resolved on each utterance.

**Why.** Without a context object, the voice layer can only interpret each utterance in isolation. That is the same trap as treating it as a CLI: "reply" has no referent, "the second one" has no list, "him" has no antecedent. Context tracking is what lets the user speak the way people actually speak: with pronouns, ellipsis, and references back to what was just said. It is the difference between voice that works and voice that feels natural.

**Implications.** `DialogContext` is a first-class type, paired with `DialogState`. Every state transition reads from and may write to the context. The audit needs to enumerate the fields the context carries: focused entity, summary slots, last utterance, last system response, recent turn history, pending confirmations. Whether context survives backgrounding the app or resets on launch is a separate decision for later.

### D4. CheckIn has an explicit persona

**Question.** Should the voice interface have a deliberately designed personality, or should we write each prompt and TTS string ad hoc?

**What the references say.** Pearl argues that every voice interface has a persona whether the designer chose one or not; the only choice is between deliberate and accidental. Google's Conversation Design treats persona as the foundation that all dialog choices reference back to: vocabulary, tone, formality, error-recovery style. Apple's HIG for Siri encourages App Intents dialog to match the app's tone consistently. NN/g notes that persona consistency is one of the strongest contributors to perceived voice-interface quality.

**Decision.** Define an explicit persona for CheckIn before writing any TTS strings. The persona statement covers default voice (locale-matched), tone, vocabulary range, formality level, verbosity baseline, error-recovery tone, and any signature phrasings. Every TTS string the app produces has the persona statement as its target.

**Why.** Without a persona, every prompt is written in isolation and the voice feels inconsistent across sessions. Users notice this even when they cannot articulate what is wrong. With a persona, prompts become testable against a clear standard, error recovery has a defined shape, and the app's voice stays identifiable across releases.

**Implications.** A persona statement is the next major design artifact after state enumeration. The statement may live as its own section in DESIGN.md or as a dedicated `PERSONA.md`; deferring that choice. Voice selection, prompt phrasing patterns, error recovery wording, and confirmation language all derive from the persona once written.

### D5. The user controls a defined set of voice parameters

**Question.** How much of the voice experience is fixed by design and how much is adjustable by the user?

**What the references say.** Pearl notes that persona sets the default but is never the only option offered to the user, and that customization respects autonomy and accessibility needs. Apple's HIG and accessibility guidelines treat speech rate and voice selection as user-controllable affordances by convention. NN/g writes that voice customization is especially important for users with hearing differences, cognitive processing preferences, or aesthetic preference; many users abandon voice interfaces over the default voice alone.

**Decision.** Expose a defined set of customization knobs in Settings. At minimum: voice selection (from the device locale's available iOS voices), speech rate, and verbosity (terse summary vs full summary). Defaults are a locale-appropriate voice and the iOS recommended rate. Customizations apply globally to all TTS the app produces. Persona shapes phrasing; user knobs shape delivery.

**Why.** Users vary widely in voice preferences and accessibility needs. A single fixed voice and rate alienates a non-trivial portion of users on first impression. `AVSpeechUtterance` supports voice and rate configuration trivially, so the cost of exposing these is low and the benefit is high. Verbosity is a useful third knob for users who already know the summary structure and want a faster review.

**Implications.** `SettingsView` gains a Voice section with the three controls. `AVSpeechUtterance` configuration reads from `@AppStorage`. The default voice is selected from the device locale's available voices, not hardcoded. The default rate is the iOS recommended value, not a hardcoded number. Verbosity affects prompt selection inside TTS generation, not the data fetched. Whether to honor system-wide accessibility speech preferences in addition to the in-app knobs is an open question for later.

### D6. Error handling is designed, not exceptional

**Question.** How does the voice interface handle recognition failures, misunderstandings, and system errors? As exceptions to be patched, or as routine flows to be designed?

**What the references say.** Pearl emphasizes that voice interfaces encounter errors constantly and that treating errors as edge cases produces a brittle, frustrating interface. Pearl recommends explicit categorization of error types (no input, no match, misrecognition, system), escalating reprompts (first gentle and rephrased, second more directive with examples, third offering a fallback or escape), and a calm error tone with minimal apology. Google's Conversation Design follows the same model: error recovery is a first-class part of every dialog, not bolted on after. Apple's HIG for Siri pushes graceful fallback to alternative modalities when voice is failing repeatedly. NN/g writes that error recovery is one of the largest predictors of voice-interface satisfaction.

**Decision.** Error handling is a first-class part of every state, not patched in later. The voice layer recognizes four error categories: no input (user did not speak within the listening window), no match (we received a transcript but could not interpret it), misrecognition (we interpreted wrongly and the user is correcting), and system error (network, auth, API failure). Each state defines its expected response to each category. Reprompts escalate: first is brief and rephrases the question, second is more directive and may offer examples, third offers a fallback (drop to touch, repeat the menu, or escape to top). Error tone follows the persona but stays calm, brief, and avoids apology spirals.

**Why.** Voice recognition is unreliable enough that every state will encounter errors regularly. Without designed error handling, the system either falls silent (worse than failing visibly) or repeats the same prompt indefinitely (frustrating users into abandonment). Designed error handling gives users a way out of any stuck state, teaches the system's vocabulary by example, and makes the app feel resilient rather than brittle. Resilience under failure is one of the strongest signals of voice-interface quality.

**Implications.** Every state in the state machine has an explicit error-handling clause covering the four error categories. The reprompt counter is part of `DialogContext` and resets on a successful turn. The state machine has an explicit escalation path with a defined termination (eventually fall to touch or top). Help-on-demand ("what can I say?") is supported as a global utterance available from any state. Error recovery wording derives from the persona statement. Misrecognition correction ("no, the other one") resolves via context lookback rather than re-prompting from scratch.

### D7. Disambiguation is an explicit pattern, not an afterthought

**Question.** When a user utterance maps to multiple possible referents or intents, how does the system resolve it?

**What the references say.** Pearl treats disambiguation as a routine voice pattern: when an utterance has multiple plausible interpretations, the correct response is to ask explicitly with a short list of options, not to guess silently. Pearl recommends presenting the options briefly and numbering them so the user can refer back ("the first" or "number two"). Apple's HIG and the `AppIntents` framework provide built-in disambiguation patterns: parameter resolution can pause to ask follow-up questions before the intent runs. Google's Conversation Design treats disambiguation as a structured turn that pauses progress until the user resolves the ambiguity.

**Decision.** Disambiguation is a first-class state, not an extension of error handling. When an utterance maps to multiple referents or intents, the system enters a disambiguation state, presents a brief enumerated list, and waits for a selection. The disambiguation prompt names the original utterance back to the user ("I heard 'Tony.' Did you mean Tony Smith or Tony Hernandez?") so they understand what is being asked. Selection accepts both ordinal references ("the first", "number two") and content references ("Tony Smith"). Context tracks what was being disambiguated so the returned answer routes back to the original pending intent.

**Why.** Voice routinely produces ambiguous referents: two contacts named Tony, two unread emails about the same project, "send" with no addressee context. Silent guessing is destructive (sends to the wrong person, opens the wrong email). Asking explicitly about everything is annoying. The middle path is structured disambiguation: the system asks only when it cannot resolve from context, asks briefly, supports an enumerated answer pattern, and confirms before any irreversible action.

**Implications.** The state machine includes an explicit `disambiguating` state with a defined entry condition (more than one candidate after context-aware matching), a prompt format, expected answer patterns, and exit transitions back to the original pending intent. `DialogContext` carries the candidate list and the suspended intent that triggered disambiguation. The intent classifier and entity matcher both expose ranked candidates with scores rather than single answers, so the dialog can decide whether to disambiguate based on score gap. Confirmation before destructive actions (send, delete, mark all read) is a related but distinct pattern, documented separately when we get to it.

### D8. The user can barge in on system speech

**Question.** Can the user interrupt TTS to start a new turn, or must they wait for the system to finish speaking?

**What the references say.** Pearl identifies barge-in (user speech interrupting system speech) as the standard term for this pattern and a major contributor to perceived responsiveness, especially for repeat users who already know what they want. Pearl recommends supporting both speech barge-in (user starts talking, system stops) and touch barge-in (user taps a control to stop TTS). Google's Conversation Design treats barge-in support as table stakes for any voice interface beyond the simplest read-aloud. Apple's `AVSpeechSynthesizer` exposes immediate stop semantics at word and immediate boundaries to support this, and iOS 17 echo cancellation on `.voiceChat` mode makes simultaneous listen-and-speak practical.

**Decision.** Both speech and touch barge-in are supported wherever the system is speaking. When the user starts speaking, the synthesizer stops at the next word boundary and the recognizer takes the foreground. When the user taps the mic button, TTS stops immediately and listening begins. When the user taps the stop control, TTS ends silently and the dialog returns to its post-speech state. Barge-in always advances the dialog rather than discarding context: the system records that it was interrupted and which content was unspoken.

**Why.** TTS is slow compared to thought. Forcing the user to wait through the full summary every time produces a sluggish interface that punishes power users and frustrates anyone who already knows the answer. Barge-in is also one of the strongest tests of whether a voice interface treats the user as a partner or a passive audience. The cost of supporting it is moderate (simultaneous audio I/O with echo cancellation, ongoing VAD during synthesis), but the cost of omitting it is paid in every session.

**Implications.** `AVAudioSession` runs in `.playAndRecord` with `.voiceChat` mode for echo cancellation while TTS is active. The audio engine input tap stays live during synthesis so VAD can detect user speech. The synthesizer stops on detection at `.word` boundary for clean cuts. `DialogContext` records the position in the spoken content where barge-in occurred and what was not delivered, so the dialog can decide whether to summarize the remainder or assume the user has the gist. The UI signals barge-in availability during TTS (visible mic indicator or waveform) so users learn the affordance.

### D9. Privacy is foundational; on-device speech recognition is non-negotiable

**Question.** Where does speech recognition happen, and how does the app handle the privacy posture for sensitive M365 content?

**What the references say.** `SFSpeechRecognizer` defaults to sending audio to Apple's servers. iOS 17 supports on-device recognition for major locales via the `requiresOnDeviceRecognition` flag. Apple's privacy guidelines emphasize on-device processing for sensitive content. UX research on voice interfaces consistently shows that perceived privacy is one of the strongest factors in adoption. Users will not use a voice interface for sensitive tasks if they suspect content leaves the device.

**Decision.** Force on-device speech recognition. `requiresOnDeviceRecognition = true` always. If the user's locale does not support on-device recognition, that is a hard preflight failure with a clear explanation, not a silent fallback to server recognition. No spoken or transcribed content ever leaves the device for recognition purposes. Privacy posture is documented prominently in the app, the README, and a dedicated `PRIVACY.md`. The open-source codebase serves as the evidence that the claims hold.

**Why.** The app processes work email, meeting subjects, and Teams chat content. Users will not, and should not, trust a voice interface that ships transcripts of work email to any server, including Apple's. Forcing on-device recognition is the only honest privacy posture for this content class. The tradeoff is locale availability: some locales lack on-device support and will be excluded at first launch. That is acceptable. A smaller, trustworthy user base is better than a larger one built on misunderstood privacy. Releasing the code as open source supports this; walking the walk requires that the code do exactly what the docs claim.

**Implications.** `SpeechService` preflights `supportsOnDeviceRecognition` for the user's locale at first launch and refuses to start if unsupported, with an explanation that names exactly which locales are supported and points to the system Language settings. The README, in-app About screen, and `PRIVACY.md` document the privacy posture in plain language: where audio is processed (on device), where text is processed (on device), what data leaves the device (only the Graph API calls the user explicitly initiates), what is logged (nothing by default; opt-in diagnostic logging only). Any future feature that would weaken this posture (cloud processing, telemetry, analytics) requires an explicit revisit at the decision level, not a quiet implementation choice.

### D10. The custom language model is opt-in, default off, with informed consent

**Question.** Should the custom language model trained on M365 contact data be enabled by default, opt-in, or absent?

**What the references say.** Apple's privacy guidelines and HIG for `AppIntents` push opt-in defaults for any feature that processes personal data beyond what is strictly required for the user's immediate task. Pearl notes the trust costs of features that quietly use personal data, even for the user's benefit. Industry norm for sensitive-data features is opt-in with clear disclosure. iOS 17's `SFCustomLanguageModelData` plus `SFSpeechLanguageModel` provides materially better proper-noun recognition than `contextualStrings` alone, but only when the user has chosen to enable it.

**Decision.** The custom language model trained on M365 contacts is implemented but disabled by default. Settings exposes a "Voice Recognition Tuning" section, off by default, with a disclosure sheet that must be dismissed to enable the feature. The disclosure names what data is used (contact display names; optional pronunciation hints), where it is used (locally, to bias the on-device speech recognizer toward the user's contacts), where it goes (nowhere; stays on device, attached to the recognizer, never transmitted), and how to remove it (single toggle that also clears the stored model immediately). The base recognition path must work acceptably without the custom model.

**Why.** Custom language model training touches contact data, which some users perceive as more sensitive than the speech audio itself. Quietly opting users in, even for a clear recognition-quality win, breaks the privacy posture established in D9. Default off with informed opt-in respects user autonomy, keeps the privacy story clean ("nothing about your contacts is read unless you turn this on"), and forces the disclosure language to be honest because users only see the toggle when they look for it. The cost is that out-of-the-box recognition for proper nouns is weaker; the off-state path must therefore use `contextualStrings` and good fuzzy matching to keep quality acceptable.

**Implications.** Settings has a Voice Recognition Tuning section, default off. Toggling on presents the disclosure sheet, then triggers a one-time language-model build from the user's address book or M365 contacts. Toggling off clears the cached model immediately. The README and `PRIVACY.md` explain the feature, what it does, and that it is off by default. Re-training on contact list changes is a separate question for later (likely a manual refresh button rather than automatic resync, to keep data movement explicit). Off-state recognition uses `contextualStrings` plus a fuzzy matcher tuned for common name variants.

### D11. No server-side speech recognition path in the app (considered and rejected)

**Question.** Should the app expose an opt-in path that uses Apple's server-side speech recognition for users who would prefer better long-form dictation accuracy and accept the privacy tradeoff?

**What the references say.** `SFSpeechRecognizer` supports both on-device and server-based recognition. Server-side has historically been stronger on free-form long-form dictation and rare vocabulary; the gap has narrowed with on-device language models in iOS 17. iOS itself exposes server-based dictation through the keyboard microphone, with system-level user controls separate from any third-party app.

**Decision.** No server-side recognition path in CheckIn. `requiresOnDeviceRecognition = true` is unconditional, with no in-app override. Users who want server-quality dictation for a long reply use the iOS keyboard microphone in the reply text field. That path exists outside the app's recognition pipeline and is governed by the user's iOS settings, not by CheckIn.

**Why.** The recognition improvement server-side would offer for this app's workload is narrow: short voice commands and short transcripts dominate, and D10's custom language model closes the proper-noun gap that matters most. The cost of an opt-in server path is large: the privacy story degrades from unconditional to conditional, two recognition paths must be maintained and audited, the custom LM cannot be used on the cloud path, and the open-source auditability of D9's privacy claim becomes harder for users to verify. The user already has access to server dictation through iOS's keyboard mic without us building anything; duplicating a system-provided affordance comes at material cost to trust.

**Implications.** No code path exists for non-on-device recognition. No setting exposes a server option. PRIVACY.md and the README state the unconditional on-device posture without caveat. If future hardware or iOS changes reopen this question (a meaningful new server-only capability becomes essential), revisit at the decision level rather than as a quiet feature add.

### D12. The Day 1 voice surface is intentionally minimal

**Question.** What set of voice interactions does the app support at launch, and what is deferred to future releases?

**What the references say.** Pearl repeatedly emphasizes that a focused, well-crafted voice surface beats a broad, shallow one. Users form their impression of a voice interface in the first few interactions, and weak performance in any one of them poisons trust in the rest. Google's Conversation Design encourages "core paths first" with explicit deferral lists. NN/g writes that voice interface adoption depends heavily on first-session success across the small set of tasks the user actually attempts.

**Decision.** Day 1 voice capabilities are deliberately minimal. The voice layer at launch supports six things: summary spoken on demand with optional sender or topic filter ("what's on my plate", "any from Tony"); refresh ("check again", "any new"); stop ("stop") and repeat ("say that again"); help ("what can I say"); voice-driven open of summary items ("open Tony's email", "open my next meeting", "open my chat with Sarah") via the deep-link path from D27; and the universal scaffolding (sign-in, settings, conversation mode entry and exit, barge-in on TTS). Out-of-scope queries are handled by D18; in-scope-but-unsupported queries are redirected to Outlook or Teams via D19 and D27. Voice features beyond this set are documented in D29 as the Day 2 and Day 3 roadmap.

**Why.** Doing a small voice surface well beats doing a broad one badly. The foundational decisions from D1 through D27 (state machine, dialog context, error recovery, disambiguation, persona, customization, barge-in, on-device privacy, custom LM with informed consent, single-screen architecture with deep-links) are not trivial to get right even for a minimal surface. Adding action operations and quick queries before the basics ship risks a launch that fails at all of them. A Day 1 that handles the summary, voice-driven open, and the surrounding scaffolding cleanly is a better introduction for users and a better foundation for the Day 2 and Day 3 expansions in D29. The iceberg framing means the architecture is sized for the full vision while the launch surface is sized for shipping carefully.

**Implications.** The state enumeration is sized for D29's full vision but the listening intent set at launch covers only the Day 1 set. The disambiguating state still applies for "open Tony's email" when there are multiple matches. Reply and email-detail have no in-app counterpart at all per D27; the touch path for both is a deep-link to Outlook or Teams. The custom language model from D10 is useful even at the smaller Day 1 scope because the summary speaks proper nouns and the user may say "open Tony's email" referencing a name they just heard; broader LM coverage matters more as Day 2 and Day 3 features land. `PLAN.md` should be updated to reflect both the Day 1 launch scope and the single-screen architecture from D27; that update is a separate task once the design audit is complete.

### D13. The app opens silent in listen mode; user initiates

**Question.** When the app opens, does it speak first (system-initiated, auto-read summary) or listen first (user-initiated, silent until asked)?

**What the references say.** Pearl distinguishes system-initiated and user-initiated voice interactions. System-initiated ("welcome to...") feels like classic IVR and tends to fight users who already know what they want. User-initiated feels more like a peer conversation: the system is ready, the user speaks first. Modern voice assistants (Siri, Alexa, Google Assistant) and modern LLM voice modes (Claude, ChatGPT, Grok) consistently use user-initiated and signal readiness with a brief earcon (audio cue) rather than a spoken prompt. NN/g and Apple's HIG note the discoverability cost of this pattern (the user has to know what to say) and recommend visible cues plus on-demand help.

**Decision.** The app opens silent and enters listen mode immediately. A short audio cue (earcon) signals "I am listening." A visible UI element (waveform or pulsing mic) reinforces the state. No canned greeting, no auto-read summary on launch. The user initiates: "what do I have today", "any new email", "summarize", "anything from Tony", or any other in-scope query. The screen still shows the full summary visually so the user has the information at a glance even if they say nothing. The voice agent behaves as a small assistant scoped strictly to the user's calendar, email, and chats. Out-of-scope queries get a brief in-character response ("I only know about your calendar, email, and chats") rather than an attempt or a generic error.

**Why.** The auto-read summary pattern was inherited from the CLI mental model, where a command produces an output and you wait for it. In a multi-modal app where the screen already shows the data, forcing TTS on every open is intrusive: it talks over users who only opened the app for a glance, and it punishes repeat use. User-initiated voice respects the user's context and matches the pattern users already know from Siri and from modern LLM voice modes. The mini-LLM framing also frees phrasing: the user asks in their own words ("what's on my plate", "anything from Tony", "any meetings before noon") rather than learning a fixed vocabulary. The earcon is the signaling mechanism: short, learnable, and does not require speech to convey.

**Implications.** The launch state is `listening`, not `summaryLoaded` plus `speaking`. A `speaking` state still exists but only as a response to a user query, never on open. Earcons need design and asset creation: at least three distinct short sounds for "listening starts," "listening ends" (response begins), and "thinking" (when a fetch is slow). Each is under 500 ms and follows iOS sound conventions. The visible listening indicator is required (D2 multi-modal): waveform or pulsing mic. The intent classifier and entity matcher must handle a wider phrasing surface than D12 originally implied: not just "check again" but any utterance meaning "give me a summary." This argues for the iOS NLP framework (`NLEmbedding`, `NLTagger`) over pure keyword matching even at launch. D12's command set is superseded: instead of a fixed verb list, the voice scope is "any query about your calendar, email, or chats" with a defined set of response shapes. Out-of-scope detection and the in-character refusal are part of the intent classification path. Help-on-demand ("what can I ask?") becomes more important under this model and must be surfaced both audibly and visually.

### D14. Voice intelligence at launch is classical NLP plus templates, not an LLM

**Question.** Does the launch voice agent use an actual LLM (Apple's Foundation Models or a bundled third-party model) for intent classification and response generation, or does it use classical NLP with templated responses?

**What the references say.** Apple's Foundation Models framework (iOS 18.1+, Apple Intelligence-capable devices only) provides on-device LLM access for developers: streaming generation, classification, summarization, structured output, tool calling. Apple's Natural Language framework (iOS 12+, all devices) provides classical NLP: `NLEmbedding` for semantic similarity, `NLTagger` for entity recognition and POS, `NLLanguageRecognizer` for language detection. Pearl notes that bounded-scope voice interfaces can be built effectively with classical NLP and good template engineering, particularly when the intent set is small and well-defined. Third-party bundled LLMs (Phi-3-mini, Gemma, Mistral via Core ML) are technically possible but add one to four gigabytes of binary size and complicate updates.

**Decision.** Launch voice intelligence is built on Apple's Natural Language framework plus a templated response generator. Specifically: `NLEmbedding` for semantic similarity scoring against a defined intent set, `NLTagger` for entity extraction (names, time ranges, action verbs), a custom intent classifier on top of these that exposes ranked candidates as required by D7 disambiguation, and a templated response generator with variable substitution and persona-shaped phrasing. No Foundation Models, no third-party LLM, no Core ML LLM bundling. Deployment target stays at iOS 17.6.

**Why.** Foundation Models would require bumping to iOS 18.1 and limiting the app to Apple Intelligence-capable devices (iPhone 15 Pro and newer). The sole known user at this point carries an iPhone 15, not 15 Pro, so the bump would exclude the user it was meant to serve. Deferring Foundation Models to the long-term backlog keeps the option open without paying its cost now. Third-party LLMs add significant binary size, complicate update flow, and weaken D9's privacy story by introducing more code surface to audit. Classical NLP with disciplined template engineering is sufficient for D13's bounded scope (calendar, email, chats), feels appropriately conversational to users, and ships on broad device support. The gap between "feels like a small assistant" and "is an LLM" is largely invisible at this scope.

**Implications.** No Foundation Models import, no third-party LLM artifacts, no Core ML LLM packaging. The intent classifier wraps `NLEmbedding` distance scoring across a defined intent set and returns ranked candidates. The entity matcher uses `NLTagger` together with the custom contact language model from D10. Response generation uses a typed template registry with persona-derived phrasings and variable substitution; no ad-hoc TTS strings outside the registry. Foundation Models adoption is added to a backlog noting the conditions that would trigger reconsideration: hardware upgrade in the user base, broader Apple Intelligence rollout, or a specific capability need that classical NLP cannot meet (open-ended summarization of long email threads, free-form rephrasing, novel intents not in the registry). `CAPABILITIES.md` should be extended with an NLP section since the original scan did not cover it.

### D15. Design for today with clean boundaries; do not pre-architect the future

**Question.** When designing components that may eventually swap to a different technology (Natural Language framework today, Foundation Models later), how much abstraction and future-proofing should we build in?

**What the references say.** The YAGNI principle ("You Aren't Gonna Need It") is widely cited in software engineering as a counter to speculative abstraction. Pearl emphasizes designing for the user need, not for technology stack ambitions. Apple's HIG and standard iOS app architecture guidance recommend protocol-based boundaries between feature code and platform APIs, but only where the boundary maps to a real present-day use case (testability, alternate implementations for specific contexts) rather than speculative future change.

**Decision.** Design CheckIn's voice intelligence layer for today's stack (classical NLP plus templates) with clean modular boundaries at the seams that already matter. Specifically: protocols for the intent classifier, the entity matcher, and the response generator, each with concrete implementations behind it. The classical-NLP implementation lives behind these protocols today; a Foundation Models implementation could be written behind the same boundaries later without rewriting callers. Do not introduce abstractions that exist only because we anticipate a future LLM migration. Do not write code that handles cases the current implementation cannot produce. Do not pre-architect interfaces around capabilities that Foundation Models has and the NL framework lacks (streaming generation, tool-calling structure, free-form rephrasing). When the migration moment comes, refactor with the actual Foundation Models surface in hand, not against guessed-at future shapes.

**Why.** Speculative abstraction has two costs: it makes today's code more complex than needed, and it almost always builds the wrong abstraction because future requirements are not actually known until the future arrives. The exception is at boundaries that already justify themselves for present-day reasons. Protocols around the intent classifier and response generator are useful regardless of LLM plans because they make the components testable, let us swap deterministic stubs for real implementations during development, and separate voice intelligence from the rest of the dialog. Other speculative abstractions (streaming token APIs, tool-call dispatch structures) build infrastructure that may not match the future need at all.

**Implications.** Voice intelligence components are protocol-typed: `IntentClassifier`, `EntityMatcher`, `ResponseGenerator`. Each protocol is shaped by today's actual needs (NL framework outputs, deterministic stubs for tests, persona-templated responses). Concrete classical-NLP implementations sit behind these protocols. No streaming protocols, no async-iteration token APIs, no tool-calling abstractions in the launch codebase. When Foundation Models adoption is later approved at the decision level, the migration starts with re-shaping the protocols around the real Foundation Models output types and writing new concrete implementations. Stub implementations stay useful for tests indefinitely. This principle applies generally beyond the LLM question: avoid speculative abstraction anywhere in the project; refactor with real requirements in hand.

### D16. Auto-listen on open is user-configurable; no wake word

*Superseded by D17. The same concern is now handled inside the broader push-to-talk vs conversation mode preference.*

**Question.** Should the app always start in listen mode (per D13), or should the user be able to choose between auto-listen on open and manual start?

**What the references say.** Pearl notes that user control over when listening begins is part of trust-building in voice interfaces, especially for apps that handle sensitive content. Apple's HIG treats microphone activation as a user-controllable affordance and does not permit third-party wake words on iOS (only "Hey Siri" can wake the device from background). Modern LLM voice modes typically require explicit user action to begin listening, citing privacy and user agency. NN/g writes that users vary widely in how comfortable they are with always-on listening, even within a single app session, depending on environment.

**Decision.** Auto-listen on open is the default but user-configurable. Settings exposes a "Listening" section with a "Voice on open" preference offering two options: "Listening (default)" and "Off, tap mic to start." When set to Listening, the app behaves per D13: opens silent, plays the listening earcon, enters listen mode. When set to Off, the app opens silent, displays the summary visually, and waits for the user to tap the mic button to begin a listening turn (the listening earcon plays at that point). No wake word in either mode. The microphone button is a visible permanent UI element in both modes; in Off mode it is the sole entry to listening, in Listening mode it doubles as a barge-in and restart-listening control.

**Why.** Auto-listen on open is the right default for a voice-first app: it matches D13's user-initiated model and removes friction for the common case. But not every user wants the microphone hot the moment they open the app. Some prefer to choose when listening begins, especially in shared spaces, on conference calls, or in private settings. Forcing auto-listen on every user is a small but real friction for these cases. Making it configurable respects user autonomy without compromising the design for users who do want the default. The "no wake word" constraint is a hard one: iOS does not permit third-party wake words from background, and any workaround using continuous foreground listening would break the privacy posture in D9 and the user-control posture in this decision.

**Implications.** Settings adds a Listening section, separate from D5's voice playback knobs, since it controls when the microphone activates rather than how speech is delivered. The launch flow reads the preference and selects between the listening-on-open path and the idle-with-mic-button path. The state machine adds an `idle` state for the Off mode (waiting for user to tap mic), which transitions to `listening` on tap. Earcons play on every transition into `listening`, so the user always knows when the mic becomes hot. No background audio entitlement is requested or used; the app does not listen when not in the foreground. The mic button's responsibilities accumulate across decisions (D2 visible mic indicator, D8 barge-in, D13 listening-state cue, D16 idle-to-listening trigger); these are coherent and should be implemented as a single multi-purpose control rather than separate buttons.

### D17. Conversation mode and tap-to-talk are both supported

**Question.** After the system finishes speaking a response, does the app return to listening automatically (conversation mode) or to idle, requiring a tap to start the next turn (tap-to-talk)?

**What the references say.** Pearl distinguishes single-turn voice interactions from multi-turn conversational ones and notes that the right pattern depends on user context. Modern LLM voice interfaces (Claude, ChatGPT, Grok, Gemini) all expose a dedicated conversation or voice mode with continuous turn-taking, separate from the simpler send-a-voice-message pattern. NN/g writes that conversation mode is preferred for hands-free contexts (driving, cooking, eyes-busy work) and tap-to-talk for shared spaces and noisy environments.

**Decision.** Settings exposes a Listening section with a single mode preference offering two options: "Tap to talk (default)" and "Conversation mode." In tap-to-talk, each turn requires the user to tap the mic; the system responds; the app returns to idle. In conversation mode, the app stays in listening between turns: after the system finishes speaking, the listening earcon plays and the mic is hot for the next turn. Conversation mode ends when the user taps a stop or close control, says a defined exit phrase ("done", "stop", "exit", "thanks"), or backgrounds the app. Every turn boundary is signaled with the appropriate earcon. Launch behavior follows from the mode: tap-to-talk opens to idle, conversation mode opens to listening. This supersedes D16.

**Why.** Tap-to-talk and conversation mode serve different user contexts, and forcing one excludes the other. Tap-to-talk suits short, deliberate queries in shared or noisy spaces and gives the user explicit control over when the mic is hot. Conversation mode suits hands-free use where multiple sequential queries are natural. The bounded scope from D12 plus the classical-NLP intelligence from D14 are sufficient for conversation mode within this app's scope, since conversation mode does not require LLM-grade understanding when each turn is independently parseable against a small intent set. The conversational feeling comes from continuous turn-taking, not from deep context understanding. The seam where this falls short is out-of-scope follow-ups, addressed in D18.

**Implications.** A single Listening preference in Settings with two options replaces D16's separate "Voice on open" toggle. The state machine has explicit `listening`, `speaking`, and `idle` states. The post-`speaking` transition is `listening` in conversation mode and `idle` in tap-to-talk mode. The audio session stays in `.playAndRecord` with `.voiceChat` continuously in conversation mode to keep the mic warm with echo cancellation. In tap-to-talk it can release the recording side after each turn for power efficiency. Earcons play at every turn boundary; conversation mode is the more demanding case because users learn the boundaries audibly. The exit-phrase detector lives inside the intent classifier and must be reliable enough that a user can always leave cleanly even under noisy recognition.

### D18. Out-of-scope responses are deliberate, varied, and in character

**Question.** When the user asks something outside the bounded scope (calendar, email, chats), how does the system respond?

**What the references say.** Pearl emphasizes that voice interfaces face out-of-scope queries constantly, especially after the rise of LLM voice modes that have no scope at all. Pearl recommends explicit scope acknowledgment with multiple phrasings, varied to avoid the scripted feel that destroys persona credibility. Google's Conversation Design treats out-of-scope as a first-class intent with its own response pool. NN/g notes that scope refusals are a major test of voice persona quality: users tolerate scope limits if the system handles them with character, and abandon if the responses sound like a chatbot dead-end.

**Decision.** Out-of-scope is a first-class intent in the classifier. When detected, the system responds with a brief, in-character acknowledgment that names the scope it does cover and invites a re-phrase. A pool of at least eight to twelve variant phrasings is maintained, drawn from the persona statement's tone, and the system rotates through them with no immediate repeats. Variants cover a range of registers: concise ("Outside my range. I know your calendar, email, and chats."), conversational ("Not my area, but I can help with meetings, messages, or chats."), inviting ("Try me on your calendar, email, or chats instead."), and so on. The system never silently fails or returns a generic error on out-of-scope input.

**Why.** Users coming from LLM voice modes expect open-domain capability and will probe the boundary, often repeatedly, in the first session. A single canned refusal repeated across these probes feels robotic and breaks the persona credibility that D4 works to build. Variation across responses mirrors how humans handle the same situation in real conversation: rephrasing, occasionally a light touch of humor, but always pointing back to what is possible. This pattern also teaches users the scope by example, which is more durable than reciting an explicit "I can do X, Y, Z" list once.

**Implications.** The intent classifier has an explicit `outOfScope` class, triggered when no in-scope intent scores above the confidence floor. The response template registry contains a dedicated pool of out-of-scope phrasings, indexed for non-repeating rotation across a session. `DialogContext` tracks the most recent out-of-scope phrasing to avoid immediate repeat. Phrasings are reviewed against the persona statement (when written) for consistency of tone. A small number of phrasings may include a light acknowledgment of the LLM expectation gap, but the pool stays mostly straightforward to keep the persona from feeling flippant. Translation to additional languages, if added later, requires a fresh pool per language since phrasings do not translate one-to-one.

### D19. In-scope but voice-unsupported queries redirect to touch with acknowledgment

**Question.** When the user asks something inside the M365 scope (calendar, email, chats) that the voice surface does not yet support (summarization of an email, reading body content, content analysis, voice reply, list browsing), how does the system respond?

**What the references say.** Pearl distinguishes "I can't help with that at all" (out of scope) from "I can't help with that yet, but here is how" (capability gap). The second pattern is gentler, acknowledges the user's reasonable assumption, and points to the working path. Apple's HIG and Google's Conversation Design both encourage explicit multimodal handoff in this situation. NN/g writes that this kind of response is one of the strongest discoverability mechanisms in multimodal apps because it teaches the user where to look without making them feel rejected.

**Decision.** "In-scope, voice-unsupported" is a first-class intent class distinct from `outOfScope` (D18). When the classifier identifies that the user is asking about a calendar event, email, or chat in a way the voice layer cannot yet handle (summarize content, read body aloud, analyze content, compose by voice, browse a list aloud), the response pattern is: a brief acknowledgment that the question is reasonable, a clear pointer to the touch path that does work ("tap the email to read it on screen"), and optionally what voice can offer instead ("or I can tell you who emailed today"). A pool of at least eight variant phrasings is maintained per sub-category (read-content, summarize-content, analyze-content, voice-reply, list-browse). The system never pretends to handle these by voice; the response is transparent about the gap and constructive about the alternative.

**Why.** This category looks like out-of-scope to a naive classifier but feels completely different to the user. "What did John say in that email" is a sensible question about content the app already shows; receiving the same generic refusal that "what's the weather" gets makes the system feel obtuse. Acknowledging the question as reasonable and redirecting to touch preserves persona credibility from D4, builds discoverability of the touch surface, and sets honest expectations about the voice scope from D12. It also paves the path for future releases: the same intent class becomes voice-supported when capability lands, with no change to what the user says.

**Implications.** The intent classifier has an `inScopeUnsupported` class with sub-categories for the major missing voice paths (read-content, summarize-content, analyze-content, voice-reply, list-browse). Each sub-category has its own variation pool keyed to the appropriate touch-path redirect. Per D27, redirects point to Outlook or Teams via deep-link rather than to internal views; phrasings such as "tap to open in Outlook" replace the earlier "tap to read it on screen." `DialogContext` tracks recent redirects to avoid immediate repeat. Phrasings are reviewed against the persona statement for tone consistency (D4) and against the verbosity knob in D5 (terse may shorten the redirect to "tap to open in Outlook"). When future releases add voice support for any sub-category (per D12's deferred-list policy), the intent class transitions from redirect to action without changing what the user says. This category is where careful light humor about the LLM expectation gap can land best, since it acknowledges the user's reasonable assumption rather than refusing it; the persona statement should set the upper bound on how far that humor goes.

### D20. UI display trusts iOS unlock; no additional content obscurance

**Question.** When the app is open on an unlocked phone, are sensitive contents (email subjects, meeting titles, chat previews) visible by default, or does the app add an additional privacy layer (auto-blur, tap-to-reveal, content hiding)?

**What the references say.** Apple's HIG treats the device-unlock state as the primary privacy boundary for app content. Most M365 apps (Outlook, Teams) display content normally on an unlocked device. NN/g notes that excessive privacy layering inside an unlocked-app context creates friction without commensurate trust gain. Pearl observes that voice-first apps do not have a meaningfully different visual privacy posture from their typed peers; the user opens the app to see and use the data.

**Decision.** No additional UI privacy layer. When the device is unlocked and the app is open, email subjects, meeting titles, and chat preview text are visible normally. The app does not auto-blur, hide, or require an additional unlock step beyond iOS. Lock screen behavior follows iOS defaults: notifications are absent (D12 chose no notifications), and the app does not bypass lock screen privacy. If the user wants additional privacy, they use iOS Screen Time, Guided Access, or close the app.

**Why.** The user opens the app because they want to see their data. Adding an additional privacy step inside an already-unlocked phone is friction without a real security gain, since anyone holding an unlocked phone has already cleared the meaningful boundary. Users would find behavior different from Outlook or Teams confusing rather than reassuring. Privacy already lives in D9 (recognition stays on device), D10 (custom LM opt-in with disclosure), D11 (no server speech), and D24 (no telemetry, no data sharing); a redundant UI layer adds nothing.

**Implications.** No screen-blur views, no tap-to-reveal interactions, no PIN or biometric step beyond iOS unlock. The app handles iOS app-switcher screenshots normally (the OS already obscures recent apps when the device is locked). No special handling for screen recording or mirroring; system-level indicators are sufficient. `PRIVACY.md` and the README state the boundary explicitly: visual privacy is the iOS unlock state, plus the audio and data postures from D9 through D11 and D24.

### D21. Latency is part of the dialog; system errors get specific response pools

**Question.** How does the system behave during normal Graph fetches (500 ms to 3 seconds), and how does it respond to specific system error categories (network down, auth expired, throttling)?

**What the references say.** Pearl emphasizes that voice latency feels longer than visual latency because the user is waiting in silence; even sub-second pauses register as "the system is stuck." Recommended pattern: an audible signal that processing is happening (earcon or short spoken placeholder), with thresholds for when to escalate to spoken reassurance. Google's Conversation Design recommends speaking only after latency crosses a perceptibility threshold (around 1.5 to 2 seconds). NN/g writes that latency tolerance in voice is roughly half of latency tolerance in GUI for the same task. Pearl distinguishes recoverable system errors (transient network, throttling) from non-recoverable ones (auth expired, account changed) and recommends different response patterns for each.

**Decision.** Latency is part of the dialog, not an interruption. The standard fetch flow is: the "thinking" earcon plays immediately on intent classification; if the fetch completes within 1.5 seconds, the response begins directly; if it exceeds 1.5 seconds, a short spoken reassurance plays from a varied pool ("checking your calendar...", "looking now...", "one moment..."); if it exceeds 5 seconds, escalate to a status update ("still working, give me a moment") and then either deliver or fail. System errors have their own response pools by category: network down ("can't reach Microsoft right now, try again in a moment"), auth expired ("your session expired, please sign in again", with the touch sign-in surface offered), throttled ("Microsoft is rate-limiting us briefly, hold on a few seconds"), unknown fallback for anything else. Each pool has at least six varied phrasings, in persona, with a clear recovery path.

**Why.** Silence during latency makes voice apps feel broken; an immediate earcon plus optional spoken reassurance keeps the user oriented. Spoken reassurance is staggered so quick fetches do not get an unnecessary "checking..." competing with the actual response. Specific error responses, rather than a single generic "something went wrong," give the user accurate information and an actionable next step. The persona stays consistent across error categories so the system feels like one assistant having a bad moment rather than a different system per failure mode.

**Implications.** The state machine has `thinking` and `speakingPlaceholder` substates inside the response flow. Earcons include a dedicated short "thinking" tone. The response template registry has separate pools for `latencyReassurance` (1.5 to 5 seconds), `latencyEscalation` (over 5 seconds), `errorNetwork`, `errorAuthExpired`, `errorThrottled`, and `errorUnknown`. `DialogContext` tracks the most recent reassurance and error phrasings to avoid immediate repeat. The auth-expired path hands off to the touch sign-in flow per D2 multi-modal; voice cannot reauth.

### D22. Voice-first does not mean voice-only; the app is fully accessible

**Question.** Does the touch and screen surface meet iOS accessibility expectations on its own, including for users who cannot speak or hear?

**What the references say.** Apple's accessibility guidelines and App Store review standards expect VoiceOver compatibility, Dynamic Type support, sufficient contrast, and reduced-motion options across all apps. WCAG informs iOS norms by analogy. Pearl notes that voice-first design is a feature on top of a fully accessible touch interface, not a replacement for one; users with speech disabilities, hearing differences, environmental constraints, or simple preference should have full functionality without speaking. Disability advocacy organizations consistently flag voice-only or voice-required interfaces as accessibility regressions.

**Decision.** The app is fully accessible to a user who cannot speak or hear. Every essential capability is reachable without voice: viewing the summary, tapping any item to deep-link to Outlook or Teams, opening Help and Settings, signing in and out. The full core experience works by touch alone. Voice-only conveniences (such as bulk operations and quick numerical queries from D29's roadmap) do not require touch equivalents inside CheckIn, since the underlying operations remain performable in Outlook or Teams via the deep-link path. The test is whether the app delivers core value without voice, not whether every voice command has a touch twin. VoiceOver labels every interactive element. Dynamic Type is honored from extra-small to accessibility sizes. Text contrast meets WCAG AA against the chosen palette. The listening waveform and pulsing mic respect Reduce Motion. The earcons from D13 are reinforced visually (waveform, indicator color change, status text) so users who cannot hear them know what state the system is in. TTS output is rendered on-screen by default (live caption pattern) so users see what the system says even when sound is off or unheard.

**Why.** Voice-first is a useful feature for many users in many contexts, but it is exclusionary if it is the only path. A deaf user cannot hear earcons; a user with a speech difference cannot reliably trigger speech recognition; a user in a noisy environment cannot hear TTS. Designing the touch surface to deliver the full core experience without voice keeps the app usable for these users, broadens the addressable audience, meets ethical obligations of accessibility-by-default, and avoids the common pattern of accessibility being added late and incompletely. Voice-only conveniences are acceptable as long as they remain conveniences, not gateways to essential function: a user who never speaks must still be able to view the summary, open items, and reach all settings.

**Implications.** Every voice path has a touch path with equivalent capability. TTS output is rendered on screen continuously when the app is in foreground. Earcons are paired with visible state changes (color, motion, text). Settings includes a "Visual reinforcement" group with options that go beyond iOS defaults (always-on captioning level, captioning persistence duration, motion intensity). VoiceOver is tested on every screen during development; missing labels are blockers, not polish items. Dynamic Type is tested at the largest accessibility size during development. The listening indicator has a non-motion variant for Reduce Motion mode. Accessibility testing is part of the pre-TestFlight checklist, not a v2 concern.

### D23. Dialog context lives in memory only; reset is natural to iOS lifecycle

**Question.** Does dialog context (focused entity, last summary, recent turn history, in-progress disambiguation) survive app close, or reset on each launch?

**What the references say.** Pearl notes that voice context survival across app close is unusual and often undesirable, since users typically expect "fresh start" semantics when they explicitly close an app. Apple's HIG for state restoration recommends restoring view state (which screen the user was on) but not implicit conversational context. NN/g writes that survival of conversational context across explicit close events surprises users in privacy-sensitive apps; reset is the safer default.

**Decision.** Dialog context lives in memory only and is never persisted. When the user closes the app (swipe up from app switcher) or iOS reclaims the app's memory, `DialogContext` is gone. When the user briefly backgrounds the app and returns, context is preserved as long as iOS keeps the app's memory intact, which is the standard iOS behavior. There is no explicit timer and no persistence layer; the natural in-memory lifetime is the boundary. User preferences from D5, D10, and D17 persist in `@AppStorage`; per D24, no M365 data is persisted, so context that references that data cannot persist either.

**Why.** Resetting on close keeps the privacy posture clean: nothing about the user's recent voice session lingers beyond the running app. Treating backgrounding as natural memory behavior matches user expectation: a user who briefly switches to another app to grab information expects to return mid-conversation, not start over. Implementing this as in-memory state rather than an explicit timer is simpler, requires no extra code, and respects iOS's own memory management.

**Implications.** `DialogContext` lives in the SwiftUI `@Observable` graph, owned by the state machine. No serialization, no Core Data, no SwiftData, no UserDefaults storage of context fields. iOS app lifecycle handles disposal naturally. User preferences are explicitly separate (preferences, not context) and live in `@AppStorage`. The reset pattern is documented in `PRIVACY.md` as part of the data-handling disclosure: conversation context is held only in the phone's memory while the app is open and is gone when the app closes or the memory is reused.

### D24. The app shares no data with anyone except the user's own M365 service

**Question.** What data does the app share, and with whom?

**What the references say.** Apple's App Privacy disclosure framework requires apps to declare the data they collect; voluntary "Data Not Collected" declarations exist for apps that genuinely do not collect anything. The Mozilla Foundation's Privacy Not Included project and the Electronic Frontier Foundation consistently flag "telemetry by default" as the most common privacy regression in apps. Pearl notes that voice apps face heightened privacy expectations because voice input is more intimate than typed input; users assume voice content stays local unless told otherwise.

**Decision.** The app contains no telemetry, no analytics, no crash reporting, and no usage logging that leaves the device. No third-party SDKs that send data anywhere. M365 data is fetched from Microsoft Graph (the user's own service, called by the user's own credentials) and processed in memory; nothing is persisted to disk, not even cached. M365 data is never sent to any third party. M365 data is never sent to Apple; the speech and NLP APIs we use (D9, D14) are on-device specifically because they do not transmit. This posture is enforced by code, not policy: the project imports no analytics SDKs, no crash reporter, and makes no `URLSession` calls except to Microsoft Graph and Microsoft login. The App Privacy declaration in App Store Connect is "Data Not Collected." The open-source repo lets anyone verify these claims.

**Why.** The privacy posture from D9 only holds if the rest of the app's data flow honors it. A voice interface that processes audio on-device but ships behavioral analytics to a cloud SDK has a contradictory posture. Promising "we don't share data" without making it impossible at the code level is a weaker commitment than promising "we have no code that could share data," because the second is verifiable. The cost of giving up telemetry is real (no production usage data, no crash logs from production users) but is acceptable: the open-source posture and the TestFlight community can fill the feedback gap by other means, and the trust gain is the entire foundation of the product's privacy story.

**Implications.** The Xcode project imports no analytics SDKs (no Firebase, Sentry, Crashlytics, Mixpanel, or equivalents). No `URLSession` calls go anywhere except `graph.microsoft.com` and `login.microsoftonline.com` (and their regional equivalents). MSAL's logger is set to silent with PII logging disabled (the MSAL default, but verified). No `os_log` or `print` statements that could leak content; logging in development uses a debug-only path stripped from release builds. App Store Connect's App Privacy section declares "Data Not Collected." `PRIVACY.md` describes the posture in plain language and notes one piece outside our control: Apple's iOS-level crash aggregation collects anonymous crash logs from all apps unless the user disables "Share With App Developers" in iOS Settings; the app cannot decline to receive these from the developer side, but the user can disable the source at the OS level. README links prominently to `PRIVACY.md`. Periodic dependency audit confirms no transitive telemetry creeps in.

### D25. Custom Azure App Registration is supported; Excelano's is the default

**Question.** Can the user override the built-in Azure App Registration with their own, and what does that look like?

**What the references say.** MSAL for iOS reads its configuration (client ID, authority, redirect URI) at startup. The client ID and authority are runtime values that can come from any source; the redirect URI must match a URL scheme declared at build time in `Info.plist`. Microsoft's Azure documentation supports the bring-your-own-registration pattern explicitly via the public client flow. Apple's HIG places advanced configuration in a separate Settings section so casual users are not exposed to it.

**Decision.** The app supports a user-supplied Azure App Registration via a Settings > Advanced section. By default the app uses Excelano's published client ID and the multi-tenant authority `https://login.microsoftonline.com/common`. Users may override the client ID and the authority URL with their own values. The redirect URI scheme stays fixed at `msauth.com.excelano.checkin`, since URL schemes are declared in `Info.plist` at build time; users running their own Azure registration must add `msauth.com.excelano.checkin://auth` as a redirect URI in their own Azure app. The Advanced section includes a "Test connection" action that attempts a sign-in cycle and surfaces specific failure modes (wrong client ID, missing permissions, mismatched redirect URI, invalid authority) with actionable guidance.

**Why.** This makes the privacy posture in D24 fully sovereign. A user who does not want any dependency on Excelano's app registration, or whose tenant governance requires apps to be registered through their own process, can run the app entirely on their own Azure infrastructure. If Excelano's registration is ever revoked, compromised, or retired, users on their own registrations continue working with no change from Excelano. Aligns with the open-source posture: anyone can verify the app's behavior and run it fully self-hosted. The cost is a moderate Settings UI addition and clear documentation; the gain is a meaningful trust and resilience improvement.

**Implications.** `Constants.clientID` and `Constants.authority` become defaults rather than hardcodes. `AuthService` reads from `@AppStorage` with the constants as fallback. Settings > Advanced exposes "Custom client ID" and "Custom authority URL" text fields with clear format hints, plus a "Why might I want this?" explainer and a link to setup instructions. A dedicated `SELF-HOSTING.md` documents the Azure setup process step by step (create registration, add redirect URI, configure permissions, enable public client, copy client ID). MSAL error responses are translated into in-app guidance for the most common misconfigurations. The redirect URI scheme cannot be customized via Settings; users who want a different scheme must build the app from source with a modified `Info.plist` (the deeper self-hosting path covered by D26). A "Reset to defaults" action lets the user clear custom values and return to Excelano's registration with a single tap.

### D26. The repository ships with step-by-step self-hosting instructions

**Question.** Beyond runtime BYO-registration in D25, can a user fully self-host the app with their own redirect URI, build it themselves, and run it independently of Excelano's published binary?

**What the references say.** Open-source iOS apps that aim for full user sovereignty (Signal, Tailscale, certain Mastodon clients) publish detailed build-and-customize instructions alongside the source. The friction of doing this without instructions is the difference between "open source as a verification mechanism" and "open source as a real fork-friendly artifact." GitHub's open-source guides recommend that any project intended to be self-hosted include explicit setup, configuration, and customization steps in the README and supporting docs.

**Decision.** The repository ships with a `SELF-HOSTING.md` document that walks through full self-hosting end to end: forking the repo, creating an Azure App Registration in the user's tenant, choosing a unique redirect URI scheme, modifying `Info.plist` and the relevant `Constants.swift` defaults to match, configuring Xcode signing to the user's own Apple Developer team, building the app, and installing it on the user's device via Xcode or TestFlight. The document is written for a competent developer who has not done iOS work before. Code structure centralizes the redirect URI scheme so the modification is a small, well-marked set of edits. README links prominently to `SELF-HOSTING.md`. The document is maintained alongside code changes; if the redirect URI configuration touches new files, the document is updated in the same commit.

**Why.** D25 covers runtime sovereignty (custom client ID and authority) without rebuild. Some users want full sovereignty: their own redirect URI scheme, their own bundle ID, their own signing identity, no shared infrastructure with Excelano whatsoever. That path requires building from source. Without documentation, the path is theoretical only; with documentation, it is real. The open-source-as-privacy-posture argument from D24 is strongest when the source can be verified and run, and making that easy is part of the posture, not extra. The cost is a one-time write of the document plus ongoing maintenance; the gain is meaningful trust and a stronger "anyone can run their own copy" claim.

**Implications.** `SELF-HOSTING.md` exists at the repo root and is referenced from `README.md` and `PRIVACY.md`. The redirect URI scheme is centralized: defined once in `Info.plist` (URL scheme entry) and referenced from `Constants.swift` so a grep for the scheme finds every location that needs to change for a fork. The document includes Azure portal screenshots where they add clarity; otherwise text-only for maintainability. Build instructions specify the Xcode version, deployment target, and any required capabilities. A short checklist section at the top gives a competent developer a path from clone to running app in under an hour. Documentation updates are a release blocker: if a redirect URI or build configuration step changes, the document must update in the same PR.

### D27. CheckIn is a single-screen voice summary; details deep-link to Outlook and Teams

**Question.** Does CheckIn maintain its own detail views, reply flow, and email body presentation, or does it deep-link out to Outlook and Teams for any work beyond the at-a-glance summary?

**What the references say.** Apple's HIG encourages apps to deep-link to first-party or trusted apps when those apps are the better tool, rather than reimplementing equivalent functionality. The Unix "do one thing well" principle applies cleanly to voice-augmented utilities: a tool that complements rather than replaces existing apps tends to be more focused, simpler to maintain, and easier for users to adopt because it does not ask them to abandon tools they already trust. NN/g writes that augmenting apps generally outperform replacing apps in the M365 ecosystem since the existing tooling has years of feature investment.

**Decision.** CheckIn is a single-screen app. The only main screen shows the at-a-glance summary: next meeting, unread emails (with senders and subjects), and pending Teams chats (with senders). Tap a meeting row and it deep-links to Outlook calendar at that event. Tap an email row and it deep-links to Outlook to that specific message. Tap a Teams chat row and it deep-links to Teams to that conversation. Voice equivalents ("open Tony's email", "open my next meeting", "open my chat with Sarah") trigger the same deep-links. CheckIn never displays email body content, never presents a reply UI, never composes. All actual mail, calendar, and chat work happens in the apps designed for it. Settings is a sheet over the main screen. Help is a sheet over the main screen. Sign-in is a modal flow before the main screen.

**Why.** This pivots CheckIn from "voice-driven email client" to "voice-first M365 status panel that augments Outlook and Teams." The simpler product is stronger: less code to write and audit (which serves D24's privacy posture), no detail or reply views to build (which collapses much of D12's deferred-voice list), no risk of users distrusting an unfamiliar reply flow with sensitive content, and a clear App Store story ("CheckIn shows your day in your voice; Outlook and Teams handle the rest"). Users already have Outlook and Teams installed; deep-linking respects their muscle memory and preserves the full feature set of those apps for actual work. The trade-off is a small reduction in control over the post-deep-link experience and a soft assumption that users have those apps installed. Both are acceptable for the M365 audience this app targets.

**Implications.** No `DetailView.swift`, no `ReplyView.swift`, no email-body parsing, no reply composition, no in-app HTML rendering. The codebase has `ContentView.swift` (auth gate), `SummaryView.swift` (the only main screen), `SettingsView.swift` (sheet), and `HelpView.swift` (sheet, defined separately). A `DeepLinkService` constructs URLs for Outlook (open inbox, open message by Internet Message ID, open calendar event by ID, open message in reply mode) and Teams (open chat by ID, open chat in reply mode, join meeting by meeting URL) and exposes a single `open(item:)` API. `Info.plist` declares `ms-outlook` and `msteams` in `LSApplicationQueriesSchemes`. `UIApplication.canOpenURL` is checked before each deep-link; if the target app is missing, the row shows a "Get Outlook" or "Get Teams" inline prompt with an App Store link. The Graph API client fetches summary fields and supports the metadata operations from D29 (mark-read, flag, soft-delete) but never fetches body content. D12's deferred voice list shrinks accordingly (see updated D12 implications). D19's in-scope-unsupported redirects point to Outlook and Teams (see updated D19 implications). `PLAN.md`'s build sequence consolidates: phase 3 becomes "single screen plus deep-link service," phases 4 and 6 (reply, voice composition) drop entirely.

### D28. Voice-initiated changes use a calibrated confirmation pattern

**Question.** When a voice command modifies or deletes data (mark read, flag, delete, bulk operations), does the system confirm before executing? When does confirmation slow the user down too much?

**What the references say.** Pearl distinguishes "always confirm" from "never confirm" approaches and prefers a calibrated middle path: confirm what is destructive or large in effect, do not confirm what is small and easily reversed. Google's Conversation Design recommends confirmation when an action affects data the user cannot easily get back; otherwise execute and announce briefly. Apple's HIG distinguishes destructive from non-destructive actions and requires destructive confirmations in touch UI; the same logic applies to voice.

**Decision.** Confirmation policy is calibrated by action class. Single mark-as-read or single flag-for-follow-up executes immediately; the system announces briefly ("marked", "flagged") with no confirmation, since both are non-destructive and easily reversed. Single soft-delete confirms ("Move that email from Bob to Deleted Items?"), since users expect a confirm step for any delete even when soft. Bulk operations of any kind always confirm with count ("Mark 4 emails from Bob as read?", "Delete 4 emails from Bob, keeping the latest?") because misrecognition cost scales with quantity. Reply by voice (deep-link to Outlook in reply mode) does not confirm; the reply is composed and reviewed in Outlook before send, where the real safeguard lives. Join meeting (deep-link to Teams) does not confirm; it is a routing action, not a modification. Confirmation prompts use the persona from D4 and draw from a small pool of phrasings per action class to avoid scripted-sounding repeats. The "yes" path executes and announces brief success ("done"); the "no" path acknowledges and returns to listening ("ok, leaving them"). A timeout in the `confirming` state without a clear yes or no returns to listening with no action taken.

**Why.** Confirming everything makes simple operations annoying and breaks the "voice as fast lane" experience. Confirming nothing risks data loss from misrecognized or misclassified utterances. The middle path is to confirm where the user could not easily recover from a mistake and to skip confirmation where the action is small and reversible. Soft-delete confirms because "delete" carries consequence even when soft (the user has to dig into Deleted Items to recover, which feels worse than an undo). Bulk operations always confirm because mistakes scale: misrecognizing "Bob" as "Tom" is a much bigger mistake when applied to four messages than to one.

**Implications.** The state machine has a `confirming` state that holds the pending action and its summary text. Entry to `confirming` plays a brief earcon distinct from `listening` so the user audibly knows the system is awaiting yes or no. Yes and no intents are universal in `confirming` regardless of what the broader intent set is. Confirmation prompts are templated with action class and parameters; persona phrasing is selected from the response template registry per D4. Timeout in `confirming` is around eight seconds to avoid a stuck state. Phrasings include sample variants per action class so identical prompts do not repeat in a session.

### D29. The voice surface roadmap (Day 2 and Day 3 expansions)

**Question.** What voice capabilities expand the surface beyond Day 1, and in what order do they ship?

**What the references say.** Pearl notes that voice interfaces benefit when the command set covers both queries (read-only information) and actions (state changes), and that staging additions after the foundation is stable lets each new capability land cleanly. Apple's HIG and Google's Conversation Design treat metadata operations on the user's own data (mark-read, flag, soft-delete) as low-risk under appropriate confirmation. The Microsoft Graph API exposes these operations as metadata-only endpoints that do not require fetching email body content, preserving the privacy posture from D9 and D24.

**Decision.** The voice surface beyond Day 1 ships in two tiers.

**Day 2 (next release after launch).** Quick queries with terse response: counts ("how many emails?", "how many from Liz?", "how many from outside my tenant?") and times ("when's my next meeting?", "how long until my next meeting?"). Mark single email as read ("mark Tony's email as read"). Flag single email ("flag Stephanie's email"). Reply by voice ("reply to Brian's email", "reply to the latest email from Stephanie") via deep-link to Outlook in reply mode.

**Day 3 (subsequent release).** Soft-delete single email ("delete Bob's email") with confirmation per D28. Bulk operations: mark-all-read, flag-all, delete-all, with the "except the latest" modifier ("delete all the emails from Bob", "mark all the emails from Bob as read except the latest one"). Bulk operations always confirm with count. Join meeting by voice ("join meeting") deep-linking to the Teams meeting if the next or current calendar event has one attached.

Time queries (Day 2) return contextual responses based on current time vs meeting start: a fixed time well in advance ("10:30"), a relative duration when closer ("in 5 minutes"), a status with offer when very close ("about to start, want to join?"), or a status with offer when in progress ("already started, want to join?"). The "want to join?" continuation is the bridge between Day 2's offer and Day 3's join action; until Day 3 ships, the offer simply doesn't appear in time-query responses.

**Why.** Day 1 ships small to test the architecture; Day 2 and Day 3 expand the surface carefully with the lessons of real use in hand. Read-only queries and easily-reversed actions land first because the cost of misrecognition is low. Destructive operations and bulk operations land second because their cost-of-misrecognition is higher, and the system benefits from proven disambiguation, confirmation, and persona consistency before users trust it with that scope. Join meeting waits until the time-query continuation is solid, since the natural entry to "join" is "want to join?" from a time query the user just asked.

**Implications.** The intent classifier expands at each tier with new categories and entity slots: counts and times (Day 2), single-action verbs and the "latest" modifier (Day 2), bulk modifiers and "except the latest" (Day 3), join-meeting (Day 3). The Graph API client adds endpoints in step: mark-read (`PATCH /me/messages/{id}`) and flag (`PATCH /me/messages/{id}/flag`) at Day 2; soft-delete (`POST /me/messages/{id}/move` to `deleteditems`) at Day 3; bulk operations (Graph batch or per-item loop) at Day 3. Reply-redirect (Outlook deep-link with message ID and reply intent) lands at Day 2. Disambiguation from D7 governs single-item actions where the sender has multiple matching messages; the explicit "latest" modifier shortcuts disambiguation. Bulk operations enumerate matching messages from the summary cache or a focused Graph query, filter by parameter, optionally exclude the latest, confirm count, then execute. Time queries derive contextual responses from current time vs the meeting's start time. External-tenant detection compares the sender's domain to the user's tenant domain, available from the signed-in account profile. The custom language model from D10 matters more at each tier because proper-noun precision drives more commands as the surface grows. Confirmation policy from D28 governs every action operation. Help content (covered in the upcoming help decision) is structured to grow per tier: Day 1 help shows the small Day 1 set; Day 2 help adds quick queries, mark, flag, and voice-reply; Day 3 help adds delete, bulk, and join. Voice-only conveniences like bulk operations and quick queries do not require touch counterparts per the updated D22.

### D30. Help system: multimodal, tapered, structured, universal, lightly contextual

**Question.** What does the help system look like: what triggers it, what does it contain, how does the user reach it, and how does it adapt to context?

**What the references say.** Pearl identifies help as a critical recovery mechanism in voice interfaces, since users frequently lose track of what is possible and need a way to re-orient mid-session without abandoning the app. Pearl recommends multimodal help (voice plus visual), tapered presentation (short by default, longer on follow-up), and contextual adaptation where useful. Apple's HIG treats help and onboarding as related but distinct: onboarding teaches first-time users; help reorients returning ones. NN/g writes that auto-help (system-launched after errors) tends to interrupt users who are still finding their footing, while passive hints (visual cues without taking over) nudge effectively without interrupting.

**Decision.** The help system is multimodal, tapered, structured, universal, and lightly contextual.

**Multimodal entry.** A permanent visible "?" button on the main screen reaches help by touch; voice "help" / "what can I say" / "what can I do" reaches the same target. Both surfaces show the same content. Voice speaks a short version; the visual screen shows the full version. Help is a sheet over the main screen, not a separate screen.

**Tapered voice.** The default voice response is short, around fifteen seconds: "I know your calendar, email, and chats. Try 'what's on my plate' or 'open Tony's email.' Tap the question mark for everything I do." Follow-up "tell me more" or "what else can I say" plays a longer version covering all Day 1 categories. The full reference always lives on the visual surface.

**Structured visual.** Three collapsible sections: "I can do this now" (the Day 1 set with example phrasings, open by default); "Coming later" (the Day 2 and Day 3 roadmap from D29, briefly noted so users see what is planned); "I don't do this" (a brief acknowledgment of D18 out-of-scope so users learn the shape of the boundary). The "I can do this now" section grows when Day 2 and Day 3 features ship; the "Coming later" section shrinks correspondingly.

**Universal intent.** Help is reachable from every state in the state machine. Returns to the prior state on dismiss: listening in conversation mode, idle in tap-to-talk.

**Auto-hint, not auto-help.** If the user hits D18 or D19 twice in a row, the visual area surfaces a discreet scope reminder ("I know calendar, email, and chats. Tap for examples."). The system does not auto-launch help. Auto-help interrupts users who are still finding their footing; auto-hint nudges without taking over.

**Lightly contextual.** Post-D18 help opens with "I can do this now" already focused (rather than the neutral structure). Post-D19 help opens with the relevant Outlook or Teams redirect category emphasized. First-time help (empty context) uses the neutral structure. Only ordering and emphasis shift; persona, content, and structure stay the same.

**Onboarding teaches help.** First-run onboarding (covered separately) ends with an explicit beat: "At any time, say 'what can I say' or tap the question mark." This makes help permanently discoverable for any user who feels stuck later.

Voice tutorial mode (system narrates three example queries with their answers) is intentionally not in this design and may never be added. The goal is for the help system above to be good enough that no user needs hand-holding beyond it. If post-launch usage data shows returning users struggling to remember the command set, voice tutorial can be added later as a separate decision; until then, it stays out.

**Why.** Help is the recovery surface for users who get stuck, regardless of whether they got there from a refusal, a misrecognition, or simple forgetting. Multimodal entry covers users who lost the voice path, and the persistent "?" button makes recovery one tap from any state. Tapered voice respects users who already know what they want and prevents the long IVR-style menu read that destroys voice-app credibility. Structured visual gives complete information without flooding the voice channel. Lightly contextual ordering uses what we already know about the user's recent state to make help more relevant without becoming a different system per refusal type. Auto-hint instead of auto-help respects user agency. Onboarding teaching help explicitly costs a single first-run sentence and pays back across every later session.

**Implications.** `HelpView` is a SwiftUI sheet over `SummaryView`. It has three collapsible sections; the initially-open section is shaped by `DialogContext` (recent D18 or D19 events). The intent classifier has a `help` class with several phrasings ("help", "what can I say", "what can I do", "what do you know"). Voice help responses are templated with a short and a long variant; the short is the default, "tell me more" extends to the long. The "?" button is a permanent UI element on `SummaryView` per D2 multi-modal. Auto-hint logic counts consecutive D18 and D19 events in `DialogContext` and surfaces a hint banner in the visual area when the count crosses two; the hint clears on the next successful intent or after a short timeout. Persona phrasings for help are reviewed against D4. Help content updates as Day 2 and Day 3 features ship; the same `HelpView` simply shows more in "I can do this now" and less in "Coming later." Help is a universal intent: it is matched in `listening`, `confirming`, `disambiguating`, and any other state, with the prior state restored on dismiss.

### D31. First-run onboarding is four steps and ends with first successful query

**Question.** What does the first-run onboarding flow look like, and what does it accomplish before the user reaches the main screen?

**What the references say.** Pearl emphasizes that voice-first apps live or die by first-run, since users have no native intuition for what to say. Recommended onboarding includes a brief introduction to scope and persona, contextual permission requests rather than a wall of system prompts, a demonstration or first-success moment, and an explicit teach of the help intent for ongoing recovery. Apple's HIG recommends requesting permissions just-in-time with context rather than at app launch. NN/g writes that users tolerate longer onboarding when each step has clear value and abandon shorter flows that feel redundant or vague.

**Decision.** First-run onboarding has four steps. Each step is a single screen with a single substantive action, and each is skippable with a safe default.

**Step 1: Welcome and privacy.** One screen explaining what CheckIn does ("voice-first daily check-in for your M365 calendar, email, and chats; opens what you ask in Outlook or Teams") and what it does not ("no body reading, no analytics, no telemetry; content stays on your device or with your own M365 service"). Inline link to `PRIVACY.md` for the full posture. Visual only; no voice yet, since speaker output before the user has chosen to engage feels presumptuous.

**Step 2: Permissions and sign-in.** Sequential prompts for microphone access and speech recognition, each with a single-line context shown before Apple's system prompt ("I listen when you talk to me. Listening only happens on your device.", "I figure out what you said using on-device speech recognition; nothing leaves your phone."). After both grants are resolved, MSAL sign-in begins. Denied permissions degrade gracefully: voice features disable but the touch path remains functional per D22. Sign-in failures retry on the same step. Once Step 2 completes successfully, the system speaks one short greeting in the persona voice ("Hi, I'm CheckIn.") to confirm speaker and voice are working and to demonstrate the persona.

**Step 3: Listening mode.** Choice between "Tap to talk (default)" and "Conversation mode" per D17, with a single-sentence description of each and a "you can change this later in Settings" line. Default selection is tap-to-talk; the user can advance with the default or pick conversation.

**Step 4: First successful query.** The system speaks one of three to five varied invitations ("Try saying: what's on my plate", "Give it a try: what do I have today?", "Ask me: any from anyone important?", and similar) drawn from a pool, then listens. The user runs their first query (the suggested phrase or any other Day 1 phrasing) and hears the response. The screen shows the summary visually as well. Below the summary, a small banner reads "At any time, say 'what can I say' or tap the question mark." This banner persists until the user dismisses it, asks for help once, or seven days have passed.

What is not in onboarding: custom language model opt-in (D10, which lives in Settings to preserve the off-by-default privacy posture), self-hosting and BYO Azure registration (D25, Settings only), and voice customization (D5, defaults work and Settings handles the rest). Onboarding does not run again on sign-out and re-sign-in: it is first-run only, gated by an `@AppStorage` flag.

**Why.** Four steps balances thoroughness and speed. Each step has one clear purpose: orientation, access, choice, first use. Skipping or denying any step still leaves the user in a usable place. Welcome plus privacy as Step 1 sets the trust posture from D9 and D24 before anything else. Permissions and sign-in together as Step 2 covers access requirements in one focused sequence. Listening mode as Step 3 is a single explicit choice that shapes interaction afterward. First successful query as Step 4 turns the abstract description into concrete experience and establishes the help affordance in muscle memory before the user is on their own. Light voice presence (one short greeting after Step 2) demonstrates the persona without speaking at the user before permissions are settled. Variation in Step 4's invitation prevents the first-run feel from being identical for every user and gives a fresh feel for anyone who clears state and re-onboards. No re-onboarding on sign-out respects users' time; they have already learned the app once.

**Implications.** A separate `OnboardingFlow` SwiftUI sequence runs once on first launch, gated by `hasCompletedOnboarding` in `@AppStorage`. The flow records the user's mode choice from Step 3 in `@AppStorage` per D17. Permission request explanations live in `Info.plist` as `NSMicrophoneUsageDescription` and `NSSpeechRecognitionUsageDescription`; onboarding screens add visual context shown before Apple's system prompt fires. The persona greeting after Step 2 uses the default voice from D5; if voice was denied, the greeting is silent and the screen advances directly. Step 4's invitation pool lives in the response template registry alongside other persona phrasings, tagged as `onboarding_invitation`. The help-teaching banner under the summary is a separate UI element that surfaces after onboarding completes and dismisses on first help use or seven-day expiry. MSAL sign-in errors during Step 2 return the user to the same step with a "try again" prompt, not to the start. Sign-out from Settings clears MSAL tokens but does not clear `hasCompletedOnboarding`; re-sign-in lands directly on the main screen.

### D32. Persona statement: calm, capable, brief; warm without familiarity

**Question.** What is the actual persona CheckIn projects in voice and text? Concretely: identity, voice, tone, vocabulary, formality, verbosity, error register, refusal register, redirect register, confirmation register, signature phrasings, and what the persona avoids.

**What the references say.** Pearl recommends a persona statement that names the character explicitly and gives the writing team concrete language for testing TTS strings against ("would CheckIn say it this way?"). Common dimensions: identity, voice, tone, formality, vocabulary, verbosity, error register, refusal register, what the persona avoids. Apple's HIG recommends voice personalities that match the app's domain and respect the user's time. NN/g writes that voice persona credibility correlates strongly with consistency: the same character across queries, refusals, errors, confirmations, and edge cases.

**Decision.** CheckIn's persona is calm, capable, and brief. Warm but not familiar. Treats the user as a competent adult who wants efficiency, not company. Light occasional dry humor on refusals and redirects; no banter on operations. The persona's name is CheckIn; there is no separate human name behind it.

**Voice.** Locale-matched system voice by default. en-US user gets a US English voice; en-GB user gets a British voice; en-DE user gets a German voice. The user can override in Settings per D5. The chosen default favors a higher-quality option when available (enhanced or neural variants) since voice quality directly shapes persona perception.

**Tone.** Calm and direct. No fillers, no preamble, no padding. Friendly enough that interactions feel pleasant; never so friendly that they feel like small talk. No "great question", no "happy to help", no apology spirals.

**Vocabulary.** Plain English. Avoids corporate jargon ("circle back", "ping", "synergy") and consumer cuteness ("yay", "awesome"). Concrete nouns and active verbs. Names items by what they are (emails, meetings, chats), not by abstractions ("items", "messages", "things").

**Formality.** Light professional. First person singular ("I", never "we"), per the user's solo-practice posture. Contractions allowed and preferred ("don't", "can't", "I'm"). No "sir", no "ma'am", no "user." Refers to the user implicitly through verbs ("you have three unread") rather than naming them.

**Verbosity.** Terse by default. The summary of "what's on your plate" is two or three short sentences. Counts return single numbers. Times return a single time or a short relative phrase. The verbosity setting from D5 expands these only when the user has explicitly opted in.

**Error register.** Calm, brief, no apology spirals. "I missed that. Try again?" rather than "I'm so sorry, I really apologize, I didn't quite catch what you said, could you possibly repeat..." Errors acknowledge briefly and reopen the path forward. Repeated errors escalate per D6 to suggest the touch path.

**Refusal register (D18 out-of-scope).** Direct, brief, names the scope. Variant pool maintained per D18; variants range from concise ("Outside my range. I know your calendar, email, and chats.") to lightly conversational ("Not my area. Try meetings, mail, or chats."). Occasional dry humor in a small fraction of variants ("I keep to your work day.") but never sarcastic, never condescending.

**Redirect register (D19 in-scope-unsupported).** Acknowledges the question is reasonable, points to the touch path, and allows a light "I'm afraid I can't do that yet" register since the user's expectation is reasonable. Phrasings come from D19's variant pools, persona-shaped. Examples: "I don't read bodies. Tap John's email to open it in Outlook." "Reading aloud isn't in my range yet. Tap to open it." A small dose of warmth fits here, since these are the moments when the user is stretching toward what feels natural.

**Confirmation register (D28).** Plain question, single beat. "Move that email from Bob to Deleted Items?" The "yes" path announces brief success ("done", "marked", "flagged"). The "no" path acknowledges and returns ("ok, leaving them.").

**Signature phrasings.** "What's on your plate" is the canonical user-side framing for the summary intent and shows up naturally in onboarding invitations and help examples. Avoid: "How can I help you today?" (too generic), "Anything else?" (chatbot dead-end), "I'm here for you!" (too eager).

**What CheckIn does not do.** Apologize repeatedly. Pad responses with politeness layers. Make small talk. Comment on the content ("looks like a busy day"). Suggest emotional reactions ("don't worry"). Editorialize about senders or meetings. Offer help in scopes it does not cover.

The persona statement above is the working reference for every TTS string in the app. Every prompt, summary, error, refusal, confirmation, and on-screen text is reviewed against it. When in doubt, the test is: would a calm, capable, brief assistant who respects the user's time say this?

**Why.** The persona is shaped by user context: working adults using CheckIn multiple times a day to triage their M365 day in voice. They do not want a friend; they want an efficient helper. They do not want banter; they want signal. Calm directness is what fits. Light dry humor on refusals and redirects keeps the persona from feeling robotic, but operations stay strictly business. First person singular respects the solo-practice nature of the project (no "we" because there is no team). Plain vocabulary and absence of corporate or consumer fluff keep the voice from sounding scripted. The "warm but not familiar" line is what separates a useful assistant from a needy chatbot; warmth gets users through misrecognitions and refusals, but familiarity wears thin across daily use.

**Implications.** A `PERSONA.md` document at the repo root captures this statement as the working artifact for TTS string writing. Every entry in the response template registry (summary phrasings, confirmation prompts, error responses, refusal pools, redirect pools, onboarding invitations, help phrasings) is reviewed against `PERSONA.md` before it ships. The persona is monitored across releases: Day 2 and Day 3 additions land in the same voice. A persona drift check is part of the pre-release checklist. A German version of the persona, if and when DE localization happens, is a parallel document (`PERSONA.de.md`), since the persona's tone in German requires native idiom rather than translation; D14's NL framework support for German covers the technical side but the persona work for DE is a separate exercise.

### D33. Application state enumeration

**Question.** What are the explicit application states the state machine traverses, what triggers transitions between them, and how do they nest?

**What the references say.** Pearl recommends explicit state enumeration as the central design artifact for any state-driven voice interface. Each state should have defined: what is showing visually, what is happening audibly, what utterances are expected, what touch interactions are valid, what dialog context is in scope, and what transitions exit the state. State machine documentation that omits these dimensions tends to produce inconsistent dialog behavior. Apple's HIG for SwiftUI app architecture aligns with state-based design as the primary organizing principle. The enumeration also serves the test suite: each state has clear pre- and post-conditions for verification.

**Decision.** The state machine is hierarchical. Top-level states are `signedOut`, `onboarding`, and `active`. The `active` state contains the operational substates the user spends nearly all their time in.

**`signedOut`.** The user has no MSAL token. Visible UI: a welcome card with a "Sign in" button. Voice is inactive. Touch action: sign in starts the MSAL browser flow. Transition: on successful sign-in, advances to `onboarding` if `hasCompletedOnboarding` is false, otherwise to `active.idle` (tap-to-talk) or `active.listening` (conversation mode).

**`onboarding`.** The first-run flow per D31 is active. Substates: `welcome`, `permissions`, `mode`, `firstQuery`. The visible UI is the onboarding sequence; the main screen is hidden behind it. Voice is inactive in `welcome` and `permissions`; the system speaks one short persona greeting after `permissions` completes; the system speaks a varied invitation in `firstQuery`. On `firstQuery` completion or skip, transitions to `active.idle` or `active.listening` per the chosen mode.

**`active`.** The signed-in, post-onboarding state. Contains the substates below.

**`active.idle`.** Tap-to-talk rest state. Visible UI: the summary plus a tappable mic button per D2. Voice is inactive. Touch actions: tap mic enters `active.listening`; tap a summary row deep-links to Outlook or Teams per D27 (transient action, returns to `active.idle`); tap "?" enters `active.helpDisplayed`; tap settings enters `active.settingsDisplayed`. `DialogContext` is empty or stale.

**`active.listening`.** Mic is hot and VAD is active. Visible UI: the summary plus a listening indicator (waveform or pulsing mic) per D2 and D13. The listening earcon plays on entry per D13. Voice action: capture user speech until VAD detects end of utterance, or a cancel, timeout, or barge-in fires. Touch actions: tap mic cancels listening and returns to rest; tap a summary row deep-links to Outlook or Teams (cancels listening); tap "?" or settings enters the displayed state (cancels listening). Transitions: on end-of-utterance, advances to `active.processing`; on silence timeout, returns to rest; on cancel, returns to rest.

**`active.processing`.** Transcript is finalized; the system is classifying intent, matching entities, and fetching from Graph as needed. Visible UI: the summary plus a thinking indicator. The thinking earcon plays on entry per D21. Substates: `thinking` (under 1.5 seconds, silent except for the earcon); `speakingPlaceholder` (1.5 to 5 seconds, plays a short reassurance from D21's `latencyReassurance` pool); `speakingEscalation` (over 5 seconds, plays a longer status update). Transitions: on response ready, advances to `active.speaking` with the response; on error, advances to `active.speaking` with the matching error response from D21; on out-of-scope or in-scope-unsupported classification, advances to `active.speaking` with the D18 or D19 response.

**`active.speaking`.** TTS is playing the response. Visible UI: the summary plus on-screen captioning of the spoken text per D22. Voice action: deliver the response via `AVSpeechSynthesizer` with delegate callbacks tracking position for D8 barge-in. Touch actions: tap mic triggers barge-in (TTS stops at word boundary, advances to `active.listening`); tap stop ends TTS silently and returns to rest; tap a summary row deep-links to Outlook or Teams. Transitions: on TTS completion in tap-to-talk, advances to `active.idle`; in conversation mode, advances to `active.listening`. Some response types (disambiguation prompts, confirmation prompts) advance to `active.disambiguating` or `active.confirming` after TTS completes rather than to rest.

**`active.disambiguating`.** The system has presented an enumerated list of candidates per D7 and is waiting for selection. Visible UI: the candidates numbered and named, a listening indicator, and the original utterance so the user knows what is being disambiguated. Voice action: listen for ordinal selection ("the first", "number two"), content selection ("Tony Smith"), date or subject reference, or cancel. Touch actions: tap a candidate selects it; tap cancel exits to rest. `DialogContext` carries the candidate list and the suspended intent. Transitions: on selection, advances to `active.processing` with the chosen candidate substituted for the ambiguous reference; on cancel or timeout, returns to rest with the suspended intent discarded.

**`active.confirming`.** The system is awaiting a yes or no for a destructive or modifying action per D28. Visible UI: the action and its parameters spelled out, a listening indicator, and a confirmation earcon distinct from listening. Voice action: listen for yes, no, or related variants. Touch actions: tap "yes" or "no"; tap cancel exits to rest. `DialogContext` carries the pending action. Transitions: yes advances to `active.processing` (executes the action); no returns to rest with the action discarded; timeout around eight seconds returns to rest with no action taken.

**`active.helpDisplayed`.** Help sheet is overlaid per D30. Visible UI: the help sheet with three collapsible sections, with the initially-open section shaped by recent context. Voice action: speak the short help variant on entry; listen for "tell me more" to expand to the long variant; listen for dismiss intents. Touch actions: dismiss returns to rest; section taps expand or collapse content. Transition: on dismiss, returns to the prior rest state (idle in tap-to-talk, listening in conversation mode).

**`active.settingsDisplayed`.** Settings sheet is overlaid. Visible UI: sections for Voice (D5), Listening Mode (D17), Voice Recognition Tuning (D10), and Advanced (D25). Voice is inactive (Settings is touch-only). Transition: on dismiss, returns to the prior rest state.

The state machine has a few universal behaviors. Help is reachable from any `active` substate via the help intent or the visible "?" button. The exit-phrase intent ("done", "thanks", "exit") in conversation mode transitions to `active.idle` regardless of substate. Sign-out from Settings transitions to `signedOut`. Backgrounding the app preserves state in memory per D23; explicit close clears it.

**Why.** Explicit state enumeration is the central artifact that pulls every other decision together. D1 said the voice layer is state-based; without an enumerated set of states, that decision is empty. Naming each state and what it permits makes inconsistencies surface during design rather than at runtime. The hierarchical structure keeps the count manageable: most operational behavior lives inside `active`, leaving the top level for major lifecycle moments. Each substate corresponds to a real user-experience moment and a real code branch.

**Implications.** `DialogState` is a Swift enum with associated values where needed (suspended intent in `disambiguating`, pending action in `confirming`, recent context in `helpDisplayed`). The state machine class owns `currentState: DialogState` and a transition function with debug-only logging. Each state has a corresponding view component or sheet. SwiftUI `@Observable` propagates state changes to the UI. Audio session configuration changes per state: `.playAndRecord` with `.voiceChat` in `listening`, `processing`, `speaking`, `disambiguating`, and `confirming`; session deactivation in `idle`, `helpDisplayed`, and `settingsDisplayed`. Earcons are tied to entry transitions: listening earcon on `active.listening` entry, thinking earcon on `active.processing` entry, confirmation earcon on `active.confirming` entry. Touch interactions on the main screen are always available regardless of voice state; voice interactions follow the state's expected utterance set. The state diagram is documented as `STATES.md` (or an appendix to DESIGN.md) for future reference.
